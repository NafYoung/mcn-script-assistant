# MCN AI 业务方向 — 实操测试最终报告

## 项目概述

本项目为 **AI Agent 解决方案实习生** 实操测试交付物。围绕虚拟品牌「轻醒」轻食酸奶的客户 brief，完成从 **小红书达人调研 → AI 工作流设计 → 脚本分镜生成 → 飞书文档自动化写入** 的全流程。

## 一、博主调研方法

### 1.1 调研路径

调研采用 **三步递进法**：

1. **平台搜索**：在小红书内以"希腊酸奶+健身早餐""酸奶碗+控糖""高蛋白+轻食"等关键词搜索相关内容创作者
2. **第三方验证**：通过新榜（newrank.cn）KOL 排行榜交叉验证博主数据（粉丝量、互动数据、内容标签）
3. **内容深度分析**：对候选博主的内容风格进行结构化拆解，评估与品牌 brief 的匹配度

### 1.2 筛选标准

| 维度 | 标准 | 原因 |
|------|------|------|
| 粉丝量级 | 5k-50k | 中腰部博主，粉丝信任度高，商单溢价空间合理 |
| 内容方向 | 轻食/健身早餐/快手菜 | 与酸奶品类天然匹配 |
| 受众画像 | 22-35岁城市女性 | 与 brief 目标人群一致 |
| 内容风格 | 分享型/教程型 | 自然种草，避免硬广感 |
| 品类关联 | 已发布过酸奶/酸奶碗相关内容 | 植入不违和 |

### 1.3 候选博主对比

| 维度 | 凌二七 | 王小饱减脂日记 | 注册营养师安娜 |
|------|--------|--------------|-------------|
| 内容方向 | 5分钟快手健康早餐 | 上班族便当/减脂餐 | 营养学科普 |
| 粉丝画像 | 22-35岁女白领 | 减脂人群 | 关注科学饮食者 |
| 品类匹配 | ★★★★★ 希腊酸奶碗为核心内容 | ★★★★ 代餐品类 | ★★★☆☆ 偏科普风格 |
| 种草力 | 强（实操教程，边做边说） | 中（偏模板化） | 弱（偏理性讲解） |
| 合规风险 | 低（无减肥叙事） | 中（减脂标签需警惕） | 低（专业身份背书） |

### 1.4 最终选择：凌二七

**核心选择逻辑**：
- 她的标志性内容是「酸奶酱」和「希腊酸奶碗」——酸奶是她内容池的原生食材，「轻醒」可直接替代她配方中的无品牌酸奶
- 简介"分享不折磨自己的健康饮食"与产品"轻负担"定位共振，天然反身材焦虑叙事
- 她从不在口播中念食品品牌名，这意味着产品可通过画面展示而非口播完成植入，不伤粉丝信任
- 2.1 万粉丝 / 13.7 万赞藏（赞藏比 6.5x），中腰部博主，商单溢价空间合理
- 已有膳美师破壁机、东菱电蒸锅等厨电品牌合作经验，观众有购买心智（评论区"求链接""什么牌子"高频出现）

**合规注意**：博主自身视频中出现了「减脂零食」「减脂期」「控糖期」「低卡」等词，商单脚本中须全部规避，改用她已在用的安全措辞（「轻负担」「好好吃饭」「能吃饱不油腻」）。

### 1.5 参考内容拆解

基于凌二七三条视频完整逐字稿的结构化拆解（详见 `references/content-breakdown.md`）：

**验证后的结构模式**（三种已验证格式）：
- **短食谱型（~60s）**：招呼 → 食材展示 → 做法① → 做法② → 做法③ → 推荐收尾
- **多品合集型（~90s）**：结果前置钩子 → 单品① → 单品② → 单品③ → 署名结尾
- **一周备餐型（~120s）**：痛点引入 → 方法论 → 单品①-⑤ → 总结公式 → 署名结尾

**固定句式（不可修改）**：
- 结尾：「我是零二七，我们一起好好吃饭。」
- 开头（二选一）：「Hello 大家，今天分享一个...」/ 「如果你也想把早餐吃得健康一点...」

**产品介绍方式（关键发现）**：
- 她从不在口播中念食品品牌名，永远描述属性（"一定要选这种浓稠的、无糖的"）
- → 商单脚本策略：口播保持原生语言，产品通过画面瓶身标签展示

**高频措辞**：「真的」「特别」「一定要」「强烈推荐」「简直完美」「不会那么噎人」「能吃饱但又不油腻」

## 二、AI 工作流设计

### 2.1 整体架构

```
客户 Brief
    ↓
[Step 1] Brief 拆解 Agent  → 结构化创作简报（含合规红线）
    ↓
[Step 2] 风格分析 Agent     → 博主风格指纹
    ↓
[Step 3] 脚本生成 Agent     → 完整脚本 + 分镜表
    ↓
[Step 4] 风险质检 Agent     → 合规报告（通过/不通过+修改建议）
    ↓
[Step 5] 飞书写入 Agent     → 飞书文档自动生成
```

### 2.2 各步骤设计要点

**Step 1 — Brief 拆解**：
- 将模糊营销 brief 转为结构化创作简报
- 核心创新：自动生成「合规红线清单」，供 Step 4 使用
- 将"卖点"映射到"用户可感知的场景"，不做成分堆砌

**Step 2 — 风格分析**：
- 提取博主"风格指纹"，包括钩子模式、语速节奏、镜头语言、口播特征
- 重点关注"该博主平时怎么说产品"，这对商单脚本最关键

**Step 3 — 脚本生成**：
- 综合创作简报 + 风格指纹，输出博主口吻的脚本
- 核心约束：口播必须读起来像博主自己写的，不能有广告腔
- 产品植入控制在 2-3 处，在"制作过程"中自然呈现

**Step 4 — 风险质检**：
- 两层检查：确定性规则扫描（硬编码禁用词列表）+ LLM 语义检查（上下文判断）
- 分离危/中危/低危三级
- 每个风险项附带具体修改建议（改写后文案）

**Step 5 — 飞书写入**：
- 格式化输出 → 调用飞书 API 自动写入
- 包含脚本正文、分镜表、质检报告、使用说明

### 2.3 核心 Prompt

所有 Agent 的 System Prompt 详见 `prompts/` 目录：
- `01-brief-deconstruct.md` — Brief 拆解
- `02-style-analysis.md` — 风格分析
- `03-script-generate.md` — 脚本生成
- `04-compliance-check.md` — 风险质检
- `05-feishu-write.md` — 飞书文档写入

### 2.4 质检规则设计

**一级禁用词（17个）**：硬编码检查，绝对不能在脚本中出现
```
减肥、瘦身、燃脂、掉秤、减脂、刮油、排油、
降糖、降血糖、降血压、降血脂、
治疗、治愈、疗效、药效、功效、
最好、第一、唯一、顶级、最强、首选
```

**二级谨慎词（5个）**：需 LLM 评估上下文后决定
```
代餐、低卡、控糖、轻断食、零脂
```

## 三、脚本与分镜设计

### 3.1 脚本概况

- **标题**：三种口味希腊酸奶，一周早餐不重样
- **博主**：凌二七（基于真实逐字稿校准）
- **时长**：~95s / 16镜
- **结构类型**：一周备餐型（参照视频3「低负担早餐必备」的已验证格式）
- **版本**：v2.0（v1.0 为推测版本，v2.0 经逐字稿校准后重写）

### 3.2 关键设计决策

**产品植入策略（v2.0 重大调整）**：
- 博主从不在口播中念食品品牌名 → 脚本中「轻醒」全程只通过画面出现（瓶身标签特写、操作中使用），口播中只说"希腊酸奶""无糖的""浓稠的"
- 画面中有 4 个产品触点：三瓶亮相(镜1-3) → 标签特写+拉丝(镜4) → 三种口味分场景使用(镜5-14) → 结尾随手拿瓶(镜15)
- 三种口味对应三种不同吃法：原味→黄瓜酸奶酱、蓝莓→酸奶碗、黄桃→荞麦拌面，自然展示 SKU 多样性

**语言风格把控（v2.0 新增）**：
- 固定结尾「我是零二七，我们一起好好吃饭」逐字锁定，不可修改
- 多处复用博主逐字稿原句（"碳水化合物蛋白质膳食纤维都有了""能吃饱不油腻""不会那么噎人"）
- 清除博主自用但违规的措辞（"减脂""低卡""控糖期"），替换为她的安全措辞（"轻负担""好好吃饭""没负担"）

### 3.3 完整脚本与分镜

详见：
- `output/final-script.md` — v2.0 完整脚本正文（基于真实逐字稿校准）
- `output/storyboard.md` — 16镜分镜表

## 四、飞书文档接入

### 4.1 接入方案

使用**飞书开放平台 API** 实现自动化写入：

1. **认证**：通过 App ID + App Secret 获取 tenant_access_token
2. **创建文档**：调用 `POST /open-apis/docx/v1/documents` 创建新文档
3. **内容写入**：通过 `POST /open-apis/docx/v1/documents/{id}/blocks/{id}/children` 分批写入（每批 ≤50 blocks）
4. **内容格式化**：使用 heading（H1-H3）、paragraph（含 bold/普通文本）、自定义分隔线等多种 block 组合

### 4.2 技术实现

- **语言**：Python 3
- **依赖**：`requests`（HTTP 调用）
- **核心文件**：`feishu/writer.py`（~330行）
- **配置**：`feishu/config.json`（App ID + Secret + 文档 Token）

### 4.3 飞书文档链接

**最终文档**：https://xcnfvm8aee87.feishu.cn/docx/IfHKdZHOWo1jOKxwQh3cv1ZbnZf

> 注：该文档由 `writer.py` 自动化创建并写入，非手动复制粘贴。

### 4.4 运行方式

```bash
# 1. 配置凭证
cp feishu/config.example.json feishu/config.json
# 编辑 config.json，填入真实 App ID 和 App Secret

# 2. 运行
python feishu/writer.py

# 3. 获取文档链接（控制台输出）
```

## 七、问题与处理记录

| 遇到的问题 | 排查过程 | 解决方案 |
|-----------|---------|---------|
| 小红书静态页面反爬 | WebFetch 返回空内容或被重定向 | 初期用新榜+搜索引擎交叉验证；后期改用小红书 App 直接浏览+录屏 ASR 转写获取真实逐字稿 |
| 脚本 v1.0 基于推测而非真实数据 | 初期内容拆解来自第三方文章转述，非一手信源 | v2.0 基于三条视频完整逐字稿重写脚本，固定句式、措辞习惯、产品介绍方式全部校准 |
| 飞书表格 API 格式复杂 | 列优先/行优先结构不明确，多次调试 | 改用结构化段落排版替代表格，保持可读性 |
| 飞书文档权限不足（1770032） | 手动创建的文档，App 无写入权限 | 改用 API 创建新文档，App 自动获得完整权限 |
| 飞书批量写入上限（50 blocks） | 单次请求最多 50 个 blocks | 实现自动分批写入（每批 50 个） |
| 飞书 bullet/divider block 不支持 | 新建文档中 block_type 11/12/18 报 1770001 错误 | 改用段落 block（type 2）+ 文本前缀（"• "、"1. "、"─"*50）模拟 |
